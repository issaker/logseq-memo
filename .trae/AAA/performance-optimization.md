3. deepseek 性能优化提示词
    - ## 任务目标：项目系统性能科学优化 (Scientific Performance Optimization)
        - 你是一位专注于 Web/客户端性能的资深工程师。请对当前项目进行**系统性的性能审查与科学优化**。你的目标是在**不破坏代码可读性、不引入过度工程化**的前提下，显著提升系统的响应速度与运行效率。
        - **核心信条**：未经测量的优化是万恶之源。必须基于数据与浏览器/框架工作原理进行优化。
        - 技术债务：请不要做向后兼容的数据设计，已经有数据迁移让用户向前更新数据格式，尽量保证系统纯净避免积累太多历史包袱。
        - ### 第一阶段：性能基准建立与瓶颈定位 (Benchmarking & Bottleneck Identification)
            - 1.  **强制前置步骤：阅读文档**：
                -   首先阅读 `README.md` 理解系统架构与核心交互流程，明确项目的**关键用户路径**（例如：首屏加载、列表滚动、数据保存、搜索过滤）。
            - 2.  **建立性能观测点**：
                -   使用浏览器 DevTools (Performance Tab, Lighthouse, Network Tab) 或框架专用调试工具，记录当前系统在关键路径上的**量化指标**：
                    -   **首屏加载时间 (FCP/LCP)**。
                    -   **用户交互响应延迟 (INP/输入反馈)**。
                    -   **内存占用与潜在泄漏 (Memory Timeline)**。
                    -   **不必要的重渲染次数 (React/Vue DevTools Profiler)**。
                -   **产出**：一份简短的**瓶颈清单**（仅在心中记录或简要笔记），明确指出当前最慢的 1-2 个核心问题。__严禁在没有数据的情况下凭直觉修改代码。__
        - ### 第二阶段：基于原理的针对性优化 (Principle-Driven Optimization)
            - 请根据第一阶段发现的瓶颈，从以下维度选择**投入产出比最高**的方案执行。**注意：若某优化手段会显著增加代码复杂度，请遵循"可维护性第一"原则放弃或寻求替代方案。**
            - 3.  **渲染效率优化 (Rendering Performance)**：
                -   **虚拟化长列表**：若存在超过 50 项的可滚动列表，**必须**评估引入虚拟滚动（或使用宿主框架原生虚拟列表组件）的必要性。原则：DOM 节点数应保持最小化。
                -   **消除无效重渲染**：
                    -   检查状态管理结构。是否因为父组件状态变更导致了子组件的不必要更新？
                    -   使用 `React.memo` / `computed` / `watch` 的细粒度特性，但要避免过度记忆化导致的代码臃肿（仅包裹开销较大的组件）。
                -   **图片与资源策略**：
                    -   检查图片是否使用了懒加载 (Native `loading="lazy"` 优先)。
                    -   检查大体积静态资源是否被合理压缩或外置。
            - 4.  **算法与数据结构提效 (Algorithm & Data Structure)**：
                -   **识别 O(n^2) 陷阱**：重点排查循环嵌套（例如：在 `forEach` 内使用 `filter` 或 `includes` 查找大数组）。
                -   **优化策略**：将数组查找转换为 `Map` 或 `Set` 索引查找 (O(1))。**修改时务必添加注释**解释为何此处使用 Map 替代 Array。
            - 5.  **网络与资源加载策略 (Network & Loading)**：
                -   **请求合并与去重**：审查短时间内发起的相同 API 请求是否有多余的重复调用。
                -   **依赖懒加载**：检查大型第三方库是否实现了按需加载（Dynamic Import），避免打包入主包阻塞首屏。
                -   **缓存策略**：对于变化频率低的数据（如配置项、主题变量），是否考虑了内存缓存层以减少重复 I/O 或网络请求？
            - 6.  **内存与垃圾回收优化 (Memory & GC)**：
                -   **事件监听器清理**：系统性检查全局事件监听 (`resize`, `scroll`, `keydown`) 和定时器 (`setInterval`) 是否在组件/模块销毁时**确定性地移除**。这是防止内存泄漏导致应用变卡的最常见源头。
                -   **闭包引用检查**：排查大型对象是否被长期持有的闭包意外引用，导致无法被 GC 回收。
        - ### 第三阶段：原生能力利用与逻辑瘦身 (Native Capabilities & Lean Logic)
            - 7.  **强制原生优先审查**：
                -   检查自定义实现的 **防抖/节流 (Debounce/Throttle)** 、**动画 (Animation)** 、**视口可见性检测 (Intersection Observer)**。
                -   **执行标准**：宿主环境（浏览器/Electron/小程序）是否已提供原生 API 或 CSS 实现？
                    -   **是** -> 移除 JS 自定义实现（哪怕只有 5 行代码），启用原生 CSS `transform`、`will-change` 或原生 Observer API。
                    -   **否** -> 保留自定义实现，但必须确保它不会与浏览器原生事件（如滚动回弹）产生冲突。
            - 8.  **计算逻辑卸载**：
                -   识别主线程中的重度计算（如大文本处理、递归遍历、加密）。
                -   评估是否可通过 **Web Worker** 转移计算任务，保持 UI 交互的丝滑。若无必要，不强行使用 Worker 增加架构复杂度。
        - ### 第四阶段：文档同步与验证 (Documentation & Verification)
            - 9.  **性能优化注释**：
                -   凡是因性能优化而改写的代码块（例如：用 `Map` 代替 `find`、添加了 `useMemo`、拆分了组件），**必须**用简短的注释说明 **"为何这样写更快"** ，防止后续维护者误以为这是冗余代码而删掉优化逻辑。
            - 10. **README 更新**：
                -   若本次优化涉及架构层面的调整（如引入了虚拟滚动库、改变了数据加载时序），需在 README 的**核心架构**或**性能注意事项**章节简要说明。
            - 11. **回归测试与性能复测**：
                -   修改完成后，必须重新运行第一阶段建立的性能观测工具。
                -   **验证标准**：
                    -   关键指标是否有**可量化的提升**（例如：FCP 降低 200ms，列表滚动帧率稳定 60fps）。
                    -   **核心功能**是否依然**100% 正常工作**（功能正确性优先于性能）。
        - ### 执行指令
            -   请先执行**第一阶段**，并向我简短汇报你识别出的**最大性能瓶颈**是什么。
            -   获得确认后，仅针对该瓶颈执行**第二阶段**的精准修复。
            -   完成修复后，清理过程中的调试日志或临时测试代码。
            -   最后，汇报**优化前后的指标对比**（即使是无工具估算的对比也可）。
            - 开始性能诊断。
