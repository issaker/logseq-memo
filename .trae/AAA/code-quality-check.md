- 任务目标：系统性代码质量检查与错误修复 (Code Quality Audit & Fix)
    - 你是一位严谨的软件工程师，负责在代码推送至 GitHub 前，对项目进行**系统性的代码质量审查与自动修复**。你的目标是通过工具辅助、静态分析和逻辑推理，发现并解决所有**功能性错误、依赖问题、代码异味和规范违背**，确保项目通过 CI 检查且长期可维护。
    - **铁律**：所有修复必须遵循项目核心原则——**可读性与可维护性优先**，严禁为了消除警告而引入不必要的复杂抽象或破坏既有架构。
    - ### 第一阶段：工具辅助的自动化检查 (Automated Tooling Check)
        - 请首先利用项目已集成的检查工具或你的内置分析能力，按以下优先级执行检查，并**收集错误与警告清单**：
        - 1.  **TypeScript / 类型系统检查**：
            -   执行 `npx tsc --noEmit` 或读取 `tsconfig.json` 配置。
            -   **重点排查**：类型错误、缺失的类型声明、`any` 滥用导致的潜在逃逸。
            -   **修复原则**：能不使用 `as` 断言就不使用，尽量通过正确的类型推导或接口补充来满足类型系统。
        - 2.  **ESLint / 静态代码分析**：
            -   执行项目配置的 ESLint 命令（如 `npx eslint .` 或 `npm run lint`）。
            -   **强制关注以下规则**：
                -   `no-unused-vars` / `@typescript-eslint/no-unused-vars`：**未使用的变量、导入、函数**。
                -   `no-undef`：使用了未定义的变量。
                -   `react-hooks/exhaustive-deps`：React Hook 依赖缺失或不稳定，可能导致闭包陷阱。
                -   `import/no-unresolved`：模块未找到或路径错误。
            -   **修复策略**：
                -   **真正的死代码**：直接删除。
                -   **暂时未用但架构预留的字段/方法**：添加明确注释说明用途，并可通过 ESLint 局部禁用注释忽略（如 `// eslint-disable-next-line no-unused-vars`），**但必须附上理由**。
                -   **依赖数组缺失**：补全依赖，或确认无依赖后用注释说明为何为空数组是安全的。
        - 3.  **依赖与包管理审计**：
            -   检查 `package.json` 与 `node_modules` 的一致性：
                -   是否存在**幽灵依赖**（引用了未在 `package.json` 中显式声明的包）？→ 必须显式安装。
                -   是否存在**未使用的依赖**（`package.json` 中声明但代码未引用）？→ 移除。
            -   执行 `npm audit` 或 `yarn audit`：关注**高/严重漏洞**。对于修复漏洞导致的大版本升级，需警惕破坏性变更；仅当安全风险明确且修复兼容时进行升级。
        - 4.  **格式化与风格一致性**：
            -   检查是否有 Prettier 或其他格式化配置。
            -   确保代码缩进、引号、分号、换行风格统一。若有冲突，以项目已有配置为准，不做主观改动。
    - ### 第二阶段：全局逻辑与架构层面的错误排查 (Global Logic & Architecture Bug Hunting)
        - 在自动检查通过后，你需要使用**架构知识**审查以下难以被工具捕获的深层问题：
        - 5.  **唯一数据流完整性**：
            -   检查是否存在同一份数据被多处分别维护，导致状态不同步的隐患。
            -   **修复**：合并到单一数据源，让所有消费方从该源读取。
        - 6.  **空值安全与边界条件**：
            -   搜索项目中所有 `?.` 可选链和 `!` 非空断言，评估其安全性。
            -   检查 `useEffect` / `watch` 等异步或响应式回调中，是否考虑了**组件已卸载**或**数据尚未就绪**的状态。必要时添加安全守卫或 `AbortController`。
        - 7.  **资源清理与内存泄漏**：
            -   审查所有 `addEventListener`、`setInterval`、`setTimeout`、WebSocket 订阅、Observer 注册。
            -   **强制要求**：每个创建资源的地方，是否在组件/模块卸载时提供了对应的**销毁逻辑**。若缺失，立即补充。
        - 8.  **异常处理健壮性**：
            -   检查 `async/await` 和 `Promise` 链是否缺失 `.catch` 或 `try/catch`。
            -   对于用户操作可能引发的错误（如网络请求失败、数据解析异常），必须有错误兜底并给用户友好反馈，而不是导致白屏。
    - ### 第三阶段：应用您的核心原则进行修复 (Principle-Driven Fixing)
        - **在修复上述所有问题时，必须严格遵循以下原则：**
        - 9.  **可读性优先**：
            -   若为了消除一个“未使用变量”警告而需要拆分组件或引入高阶函数，导致可读性下降，则**宁可利用注释禁用该警告**，也不破坏代码的直白性。
            -   修复 Bug 时，优先选择最简单、最符合人类直觉的写法。
        - 10. **原生优先**：
            -   如果发现用来解决某问题的第三方库，新版本的宿主环境（浏览器/Electron/Node）已经原生支持，则**趁此机会移除该依赖**，改为原生实现，并清理相关引用。
        - 11. **拒绝过度工程化**：
            -   修复类型错误时，不要为了完美类型而引入高度抽象的类型体操（如复杂的泛型递归）。够用、准确即可。
            -   不要因为扫描出 1 个边缘情况，就引入一套全局错误处理中间件，除非项目已接近该架构范式。
    - ### 第四阶段：修复后的验证与文档同步 (Verification & Doc Sync)
        - 12. **回归验证**：
            -   修复所有错误后，重新运行一遍第一阶段的自动化检查命令，确保**零错误零警告**（故意保留且注释说明的警告除外）。
            -   快速手动过一下项目的**关键用户流程**（启动、核心操作），确保修复没有引入回归。
        - 13. **清理与文档**：
            -   删除修复过程中产生的临时文件、测试脚本。
            -   如果修复涉及架构调整（例如：移除某个依赖改原生实现、改造某数据流），请更新 `README.md` 中对应部分，保持文档与代码的一致性。
        - 14. **输出修复摘要**：
            -   最终向我简要汇报修复的问题数量与类别，例如：
                -   移除未使用变量：3 处
                -   补充缺失依赖：1 个包
                -   修复内存泄漏风险：2 处（事件监听未卸载）
                -   禁用合理保留的未使用变量警告：1 处（附注释）
    - ### 执行指令
        -   请按**第一阶段 → 第二阶段 → 第三阶段**顺序执行。
        -   第一阶段的每个子步骤完成后，向我展示关键错误清单。
        -   第三阶段修复时，若遇到需要权衡的复杂情况（如需过度抽象），暂停并向我说明，获得确认后再继续。
        -   最终必须运行一次完整检查，确认全部通过。
        - 开始代码质量审查与修复。
