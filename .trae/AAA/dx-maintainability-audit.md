2. 任务目标：开发体验与可维护性审查 (DX & Maintainability Audit)
    - 你是一位 Developer Advocate 兼资深工程师。请以"一个完全不了解本项目的开发者首次接触该仓库"的视角，审查并优化项目的上手体验和长期可维护性。
    - ### 1. 五分钟能否跑起来？
        - 按 README 的安装步骤执行一遍，遇到任何卡顿即视为 Bug：
            - Node 版本要求是否明确？
            - `npm install` 是否一次性成功？是否有 peer dependency 冲突？
            - `.env.example` 是否存在且包含所有必要字段？
            - 是否需要额外全局工具（如特定的 CLI 版本）但未声明？
        - **修复所有导致无法"开箱即跑"的障碍。**
    - ### 2. 脚本与工作流清晰度
        - 审查 `package.json` 的 `scripts`：
            - `dev` / `build` / `test` / `lint` 是否齐全且命名符合社区惯例？
            - 是否有脚本负责数据迁移、数据库初始化等启动前置任务？
        - 补充缺失的关键脚本，并确保 README 说明了每个脚本的用途。
    - ### 3. 环境切换安全性
        - 检查是否有开发 / 生产 / 测试环境的隔离机制。
        - 确保不会意外将测试脏数据写入生产数据库，或反之。
    - ### 4. VSCode / IDE 配置共享
        - `.vscode/settings.json` 和 `extensions.json` 是否存在？
            - 格式化器 (Prettier/ESLint) 配置是否统一且自动生效？
            - 推荐的插件列表是否让新成员一键安装？
        - 若缺失，创建之。
    - ### 5. .gitignore 审计
        - 是否遗漏了常见的应忽略文件：
            - `.env` 文件（防止密钥泄露）
            - 构建产物 (`dist`, `build`, `out`)
            - OS 文件 (`.DS_Store`, `Thumbs.db`)
            - IDE 个人配置（但保留共享配置）
            - `node_modules`
        - 逐一比对并补全。
    - ### 6. 输出 DX 改进摘要
        - 列出本次修复的所有改善点，让团队知道"新成员上手体验提升了多少"。
