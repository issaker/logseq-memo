6. 任务目标：项目终态清理与极致精简 (Final Cleanup & Minimalist Delivery)
    - 你是一位极致主义的工匠工程师。本项目已经历了代码质量检查、开发体验优化、性能调优、架构重构和测试补强，现在是最后一步：**彻底清理一切非必要资产，确保交付的代码库像一把擦亮的手术刀——没有一克多余的脂肪。**
    - **核心信条**：代码库的纯洁性直接影响长期维护成本。每一行无用的代码、每一个多余的字节，都是对后续维护者心智的消耗。
    - ### 第一阶段：死代码与冗余逻辑清理 (Dead Code & Redundancy Elimination)
        - 1.  **无引用的代码孤岛**：
            -   **强制工具检查**：重新运行 ESLint 的 `no-unused-vars`、`no-unused-imports` 规则（或 `ts-prune` 工具），识别所有未被引用的变量、函数、类型、接口。
            -   **组件/模块级检查**：查找项目中**从未被导入**的 `.ts`/`.tsx`/`.vue` 文件。删除它们。
            -   **注释掉的代码块**：搜索所有被注释掉的代码（`// old code`、`/* ... */` 包裹的旧实现）。除非有明确标注为“保留用于应急回退且近期可能启用”的注释，否则**一律删除**。Git 历史已是最好的备份。
        - 2.  **向后兼容代码的彻底清理**：
            -   **硬性指令**：项目原则中已明确“不做向后兼容”。请全局搜索关键字 `compat`、`legacy`、`oldVersion`、`deprecated`、`v1`、`v2` 等，清除所有仅为兼容旧版数据结构或 API 而保留的分支逻辑。
            -   确认 `Data Migration` 模块已能处理所有旧版数据升级，之后**将兼容代码连根拔起**。
        - 3.  **重复逻辑的精简**：
            -   在不违反“可读性优先”和“模块独立性”原则的前提下，合并功能完全相同的工具函数。
            -   **警惕过度清理**：如果两个函数代码相似但服务于明确不同的业务模块，且合并会导致耦合或增加理解成本，则保留独立。
    - ### 第二阶段：文件系统瘦身 (Filesystem Slimming)
        - 4.  **临时与测试文件清理**：
            -   搜索并删除所有**一次性脚本**：如 `fix-data.ts`、`migrate-local.ts`、`test-*.js` 此类仅用于某次手动操作的临时文件。
            -   删除所有**临时测试文件**：与核心测试套件无关的 `__tests__/scratch.test.ts`、`sandbox/`、`playground/` 等。
            -   删除**构建调试产物**：`sourcemaps` 中的无关文件、未清理的 `stats.html` 等。
        - 5.  **文档资产精简**：
            -   搜索项目根目录及子目录下的 `.md` 文件，除 `README.md` 和 `THEME_SYSTEM.md` 等核心文档外，常见的待清理文件：
                -   `NOTES.md`、`TODO.md`（其重要内容应转入 GitHub Issues，否则删除）
                -   `CHANGELOG_old.md`、`MIGRATION_GUIDE_v1.md`（过时的迁移指南）
                -   `REFACTOR_PLAN.md`、`DESIGN_DOC.md`（若其内容已实现并融入 README，则删除）
            -   **清理策略**：对于任何非核心文档，如果其包含的**关键信息**未被 README 覆盖，则**提取关键信息补入 README 的对应章节**，然后删除该文件。不留孤立的说明文档。
        - 6.  **资源文件审计**：
            -   检查 `assets/`、`images/`、`public/` 中是否有**未被任何代码引用的**图片、字体、样式文件。
            -   识别并删除重复或过时的设计稿、图标草图。
    - ### 第三阶段：注释与说明的压缩提纯 (Comment Compression & Purification)
        - 7.  **冗余注释清理**：
            -   **删除重复代码本义的注释**：例如 `// set user name to input value` 紧跟着 `user.name = input.value`，此类注释零信息量，一律删除。
            -   **删除被注释掉的死代码**：如 `// oldMethod()` 这类僵尸代码，已在第一阶段清理，此处确保无残留。
            -   **保留并强化“为什么”的注释**：任何解释**业务规则、非直观逻辑、性能考量、临时权衡**的注释，予以保留，并确保其表述清晰准确。
        - 8.  **注释风格统一**：
            -   确保注释语言一致（中英统一，建议英文）。
            -   函数/类的 JSDoc 注释只保留对外暴露的公共 API，内部私有函数若无复杂逻辑则无需装饰性 JSDoc。
    - ### 第四阶段：全局复查与最终确认 (Global Review & Final Confirmation)
        - 9.  **全局引用完整性检查**：
            -   运行完整的测试套件（`npm test`），确保删除操作没有破坏任何功能。
            -   再次运行 Lint 和类型检查（`npm run lint && npx tsc --noEmit`），保证零错误零警告。
        - 10. **Git 状态验证**：
            -   检查 `git status`，确保所有删除的文件都在暂存区。
            -   建议运行 `git diff --stat` 查看变化概览，确认没有误删关键文件。
            -   **特别注意**：`.gitignore` 中应列出所有不纳入版本控制的文件类型，以防未来再被提交。
        - 11. **最终交付品清单**：
            -   输出一份简洁的清理报告：
            - 删除未引用文件：X 个
            - 删除死代码块：Y 处
            - 合并冗余工具函数：Z 个
            - 移除过时文档：N 个（已提取关键信息至 README）
            - 精简冗余注释：M 行
            - 项目体积减少：约 XX KB
        - ### 执行指令
            -   请按第一阶段 → 第二阶段 → 第三阶段顺序执行。
            -   每一个删除动作前，你可以先在脑中或向我简略提问确认涉及文件数量较大的操作（如删除整个目录）。
            -   最终必须保证 `npm test` 和 `npm run lint` 全部通过。
            -   完成清理后，向我提交清理报告。
            - 开始项目的终极瘦身。
